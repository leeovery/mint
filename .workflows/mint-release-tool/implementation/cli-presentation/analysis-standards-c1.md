- FINDING: Plain blocking-stage start line hardcodes "generating..." for every blocking stage
  SEVERITY: medium
  FILES: internal/presenter/plain.go:147-152
  DESCRIPTION: StageStarted (plain) renders a fixed `"{name}: generating...\n"` for ALL blocking stages. The spec's event-payload principle (specification.md lines 64-66) requires the engine to supply "every datum the renderings consume" and states "the presenter never re-derives engine knowledge or hardcodes stage-specific logic." "generating" is stage-specific narration: it fits the AI notes stage (the spec's only worked example, `notes: generating…`) but reads wrong for the other named blocking stage — the `pre_tag` build hook (`prep`), which the spec explicitly calls long/blocking (lines 66, 208). That stage renders `prep: generating...`, which is semantically incorrect (a build hook is not "generating"). The StageStart payload carries only Name + Blocking, so the presenter has no engine-supplied start verb to render and must invent one. The spec grants "wording refinable" latitude, so this is not a hard contract break, but the single hardcoded verb is exactly the presenter-side stage assumption the payload principle is designed to prevent and it produces wrong-sounding output for a stage the spec names.
  RECOMMENDATION: Either neutralise the synthesised word so it is stage-agnostic (e.g. `"{name}: running...\n"`, correct for both notes generation and a build hook), or carry an engine-supplied start detail on StageStart and render that (keeping the engine as the source of the text per the payload principle). The pretty side already sidesteps this by rendering only the dim stage name as spinner text.

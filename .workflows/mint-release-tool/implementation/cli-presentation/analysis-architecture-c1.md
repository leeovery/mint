- FINDING: Pretty constructor surface has four overlapping public entry points with ad-hoc capability combinations
  SEVERITY: medium
  FILES: internal/presenter/pretty.go:183, internal/presenter/pretty.go:194, internal/presenter/pretty.go:203, internal/presenter/pretty.go:214, internal/presenter/pretty.go:272
  DESCRIPTION: PrettyPresenter exposes four exported constructors (NewPrettyPresenter, NewPrettyPresenterWithProfile, NewPrettyPresenterWithErr, NewPrettyPresenterWithInput) plus a WithInput setter that exists specifically to patch a gap one constructor leaves (NewPrettyPresenterWithErr defaults input to os.Stdin, so WithInput re-injects a scripted reader). Each constructor hard-codes a different subset of the three orthogonal knobs (profile, err writer, input reader), so the matrix of "force colour AND capture err AND script input" is only reachable by chaining a setter onto a constructor that omits one axis. This is the constructor-explosion the field docs themselves anticipate ("kept off the constructors so their signatures stay stable... without a constructor explosion") but the explosion happened on the constructor side instead. The skill packs call for functional options precisely here (golang-design-patterns: "Constructors SHOULD use functional options - they scale better as APIs evolve"). The plain side stays clean with two constructors plus chainable setters; pretty's four-constructor surface is the asymmetric outlier and the one place a future fourth axis forces yet another NewPrettyPresenterWith... variant.
  RECOMMENDATION: Collapse the three profile/err/input variants into one constructor taking functional options (e.g. NewPrettyPresenter(out, opts ...Option) with WithProfile/WithErr/WithInput options), keeping a thin production entry. This removes the WithInput-patches-a-constructor-gap wart and makes every capability combination expressible without a bespoke constructor. Pure test-ergonomics refactor; the rendered behaviour and the Presenter seam are unaffected.

- FINDING: SourceGate/TargetGate reuse the Choice/GateChoice vocabulary built for fixed semantic answers, conflating two kinds of "choice"
  SEVERITY: low
  FILES: internal/presenter/gate.go:184, internal/presenter/gate.go:200, internal/presenter/gate.go:9, internal/presenter/prompt.go:47
  DESCRIPTION: Choice is documented as "the ENGINE's vocabulary - accept, abort, edit, regenerate" - a closed semantic set (y/n/e/r). SourceGate/TargetGate reuse that same Choice type to carry open-ended enumerated values (github/gitlab, stable/beta) as GateChoice keys, and set AcceptEcho = string(def). This overloads one type across two semantically distinct roles: a fixed four-member answer vocabulary versus an engine-supplied data enumeration. The reuse works and lets one Prompt method render every gate (a genuine win), but the shared parse path lower-cases and case-folds source/target values the same way it folds y/n (parseChoice), and the AcceptEcho byte-purity constraint ("must stay byte-pure ASCII for the plain echo") now silently depends on every engine-supplied source/target value being ASCII - a contract the type does not enforce and that would render wrong (or trip the plain byte-purity guard) for a non-ASCII target name. The seam holds for the current enumerations but rests on caller discipline rather than being self-contained.
  RECOMMENDATION: No structural change required now - flagging the latent assumption. If source/target values ever broaden beyond ASCII enumerations, the case-folding parse and the ASCII-only AcceptEcho become wrong; a comment on SourceGate/TargetGate stating the "ASCII enumerated values only" precondition (or a constructor-time guard) would convert the silent dependency into a stated contract.

SUMMARY: Architecture is sound overall - a clean dependency-inversion Presenter seam, well-typed payload structs (no untyped containers), a shared parse/loop composed across both modes, useful zero values, exhaustive enum dispatch, and a single wiring point with the stream split established in one place. The two findings are a medium-severity test-constructor proliferation on the pretty side (functional options would flatten it) and a low-severity latent ASCII/case-fold assumption from reusing the Choice vocabulary for enumerated source/target values.

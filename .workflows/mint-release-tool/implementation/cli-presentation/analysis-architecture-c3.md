- FINDING: Startup-wiring paths have diverged — the fully-resolved signal struct is unconsumed and the production constructor bypasses both gating axes
  SEVERITY: medium
  FILES: gating.go:38-64, wiring.go:47-53, plain.go:99-114, pretty.go:267-282
  DESCRIPTION: The package grew three overlapping startup constructs that do not converge into one fully-wired seam. `DetectStartupSignals` returns a `StartupSignals{Mode, StdinInteractive}` that resolves BOTH orthogonal axes once — but nothing consumes it (only its own test references it). Meanwhile `NewForStartup`, documented as "the default startup convenience" / "the one construction site", calls `DetectMode` directly and threads only Mode + terminal width; it never threads `-y` (`WithYes`) or the stdin-interactive signal (`WithInteractiveStdin`). The two gating signals are reachable only via builder setters whose doc comments defer them to "a later main/cmd task". The net effect is two parallel startup paths that have already drifted: the struct designed to carry the gating axis (`StartupSignals.StdinInteractive`) is wired to nothing, while the actual production constructor cannot produce a presenter that reaches the forbidden-combination fail-loud path (a load-bearing spec requirement: non-TTY stdin without `-y` must fail loud) because it leaves both gating fields at their interactive defaults. An engine author wiring this package faces an ambiguous surface: `DetectStartupSignals` looks like the resolution entry point, `NewForStartup` looks like the construction entry point, and they neither compose nor agree on which axes they handle. This is a seam-quality gap at the package's outermost boundary.
  RECOMMENDATION: Make `NewForStartup` (or a single new entry point) the one seam that consumes `StartupSignals` and threads all three signals — Mode, width, `-y`, and `StdinInteractive` — onto the returned presenter, so the gating axes are wired at the same site as the render axis rather than left to undocumented downstream setters. Keep it within the presenter package (no main/cmd). One converged startup path removes the ambiguity and guarantees the fail-loud path is reachable through the documented startup seam.

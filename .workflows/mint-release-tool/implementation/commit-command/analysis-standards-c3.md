- FINDING: `[commit].context` lacks the string-or-file resolution that `[release].context` provides
  SEVERITY: low
  FILES: internal/commit/prompt.go:109-122, internal/commit/prompt.go:128-130, internal/config/config.go:546-557, internal/notes/prompt.go:120-167
  DESCRIPTION: The spec (Commit Message Format & Prompt) describes commit's two prompt knobs as a "Two-knob override, mirroring release ... Same 'mint owns the prompt; ai_command is transport' model." Release's context knob (`internal/notes/prompt.go` resolveContext) treats `[release].context` as "string OR file": if the value names an existing file under root, the file's contents are injected; otherwise it is a literal string. Commit's `ResolveInstructions`/`injectContext` always treats `cfg.Commit.Context` as a literal string with no file detection. A user who, by analogy with release, points `[commit].context` at a file path would get the path string injected verbatim instead of the file contents. This is a cross-verb behaviour divergence in a feature the spec says "mirrors release" — exactly the kind of drift a single-slice executor (who never saw release's context resolver) would not catch. NOTE: the spec's `[commit].context` example shows only a literal string and never explicitly pins file-detection, so this is a parity/convention concern, not a hard spec violation; commit's full-override knob (`[commit].prompt`) does correctly mirror release's file-read-and-fail-loud behaviour.
  RECOMMENDATION: Either (a) make `[commit].context` apply the same string-or-file detection as `internal/notes`' resolveContext for true release parity, or (b) if literal-only is the deliberate intent, add a one-line code comment in prompt.go documenting that commit's context is literal-string-only (unlike release) so the divergence is an explicit, recorded decision rather than silent drift.

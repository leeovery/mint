- FINDING: Preflight emptiness probe and L1 generate source are parallel, independently-authored implementations of the same per-mode source selection
  SEVERITY: medium
  FILES: internal/commit/run.go:715-749 (wouldStageNothing, stagedProbeArgs, trackedProbeArgs, untrackedProbeArgs), internal/commit/generate.go:144-252 (sourceDiff, stagedDiff, trackedWorktreeDiff, addAllWorktreeDiff, untrackedFiles)
  DESCRIPTION: The "something to commit" preflight and the L1 message-generation source both compute the would-be-staged set for the same StagingMode, from argv that must stay byte-for-byte aligned. They are two separate `switch g.mode` / `switch mode` statements with two separate sets of argv builders: the probe builders (`{"diff","--cached","--name-only","--","."}` + excludes, etc.) and the source helpers (`{"diff","--cached","--","."}` + excludes, etc.) differ only by `--name-only` yet each re-spells the source-command prefix and the `-- .` selector. The spec makes "the preflight and the AI's L1 diff read ONE exclusion-filtered source and cannot drift" a load-bearing invariant, but correctness here rests on a human keeping two parallel argv builders in lock-step — exactly the parallel-computation-that-must-be-kept-in-sync pattern. The shared `excludePathspecs` helper correctly factors the exclusion tail, but the source-prefix + selector are duplicated. The emptiness verdict is logically derivable from the source (empty iff the source diff is empty), so the probe could be expressed in terms of the L1 source selection rather than re-deriving it. (Note: the AddAll probe deliberately avoids rendering each untracked file's body for cost, so the perf rationale is real and partly justifies a lighter probe — but the *source-selection logic*, "which commands does this mode read", need not be duplicated.)
  RECOMMENDATION: Factor the per-mode source-command selection into one place both consumers share — e.g. a single helper that, given a StagingMode, returns the ordered (command-prefix) building blocks, so the probe argv is derived by adding `--name-only` to the same prefix the source helper uses, and neither the prefix nor the `-- .` selector is spelled twice. This makes the "one source, cannot drift" invariant structural rather than convention-enforced.

- FINDING: commitReviewGate fully re-declares NotesReviewGate to vary only Subject
  SEVERITY: low
  FILES: internal/commit/run.go:633-646 (commitReviewGate), internal/presenter/gate.go:143-159 (NotesReviewGate)
  DESCRIPTION: commitReviewGate() is byte-identical to presenter.NotesReviewGate() except for Subject ("message" vs "notes") — the same Question ("Continue?"), AcceptEcho ("accepted"), Default (ChoiceYes), and the same four-choice y/n/e/r Choices slice are hand-copied. The documented reason for not importing directly (the -y echo must read "message: accepted", not "notes: accepted") is valid, but the response re-declares the entire gate to override one string field. Gate is a plain value type, so the choice vocabulary could be derived (take NotesReviewGate(), set Subject) instead of duplicated. As written, if the shared gate's choice set evolves (a new action key), commit's hand-built copy silently drifts out of alignment with the notes gate it mirrors.
  RECOMMENDATION: Derive commit's gate from the shared constructor and override only the differing field (e.g. `g := presenter.NotesReviewGate(); g.Subject = "message"; return g`), or add a small parameterised constructor on the presenter that takes the Subject. Either keeps the y/n/e/r choice vocabulary single-sourced while preserving commit's distinct Subject.

- FINDING: cmd layer double-detects startup TTY signals because the Presenter exposes no StdinInteractive accessor for the engine-layer editor guard
  SEVERITY: low
  FILES: cmd/mint/main.go:293-318 (runCommit: DetectStartupSignals + NewForStartup), internal/commit/run.go:212-243 (Deps.StdinInteractive, editorUnavailable)
  DESCRIPTION: runCommit calls presenter.DetectStartupSignals(...) solely to extract StdinInteractive as a boolean to thread into commit.Deps, then calls NewForStartup(...) which internally re-runs the identical stdout/stdin TTY detection. The editor-fallback no-message-source guard (Deps.editorUnavailable) needs this signal at the engine layer — before any presenter Prompt is issued — but the Presenter interface deliberately encapsulates the same determination for the gate's own forbidden-combo and offers no accessor, forcing commit to re-derive it. Engine/release never needs this (its forbidden-combo is fully presenter-internal), so commit is the first consumer to surface the gap. The two detections are pure over the same descriptors and cannot disagree in practice, so this is a contained seam wrinkle, not a correctness risk — but it is a parallel detection of a signal the presenter already computes.
  RECOMMENDATION: Have the Presenter (or its startup-signal value) expose the resolved StdinInteractive so the engine-layer guard reads the single computed signal, eliminating the cmd-layer's parallel DetectStartupSignals call. Low priority — acceptable as-is given both probes are pure and read the same descriptors.

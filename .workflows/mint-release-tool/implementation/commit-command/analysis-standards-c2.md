- FINDING: [commit].context lacks the string-or-file affordance [release].context has
  SEVERITY: low
  FILES: internal/commit/prompt.go:109-122, internal/config/config.go:459-461, internal/notes/prompt.go:120-168
  DESCRIPTION: The spec frames commit's two prompt knobs as "Two-knob override, mirroring release ... Same 'mint owns the prompt; ai_command is transport' model" (Commit Message Format & Prompt; Config Schema). Release's [release].context knob is "string OR file": notes.resolveContext stats the value and, if it names a real file under root, injects the file's CONTENTS; otherwise it treats the value as a literal string. Commit's ResolveInstructions instead injects cfg.Commit.Context verbatim as a literal string only (no file detection) — so a [commit].context that names a file path injects the path text rather than the file body. This is a behavioural asymmetry against the "mirroring release" framing. It is LOW severity because the spec's commit config example and prose only ever show an inline literal context string ("Conventional Commits; dev-workflow toolkit.") and never explicitly require file-loading for commit context, so the literal-only reading is defensible — but the two context knobs now diverge for the same documented concept.
  RECOMMENDATION: Either (a) reuse the same string-or-file resolution release uses for [commit].context so the two knobs behave identically (aligning with the "mirroring release" intent), or (b) if literal-only is the deliberate choice, add a one-line note to the spec/Config Schema recording that [commit].context is inline-string-only (unlike [release].context) so the asymmetry is documented rather than incidental.

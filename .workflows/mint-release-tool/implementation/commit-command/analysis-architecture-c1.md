- FINDING: Empty-staging preflight and L1 source diff are parallel "would-be-staged" computations that drift on diff_exclude — AI can be invoked on an empty diff
  SEVERITY: high
  FILES: internal/commit/run.go:697-713 (wouldStageNothing), internal/commit/generate.go:144-174 (sourceDiff/stagedDiff), internal/commit/generate.go:108-131 (GenerateWithContext), internal/notes/size.go:34-40 (CheckDiffSize)
  DESCRIPTION: Two independently-authored queries answer the same question ("is there anything to commit?") but apply DIFFERENT filters. The preflight emptiness probes (wouldStageNothing) run name-only git commands WITHOUT the diff_exclude :(exclude) pathspecs — e.g. `git diff --cached --name-only`. The actual L1 content the AI sees (sourceDiff/stagedDiff/trackedWorktreeDiff/untrackedFiles) is built WITH excludePathspecs(cfg.DiffExclude). When every staged/changed file matches a diff_exclude glob, preflight sees a non-empty set and passes, then sourceDiff returns an empty post-exclusion diff. CheckDiffSize("",n) returns nil (countDiffLines("")==0), the transport validates only the AI's RESPONSE not its input, and there is no empty-post-exclusion guard anywhere in the generate path — so transport.Generate is called with "instructions\n\n" (a blank diff). This directly violates the spec invariant "never invoke the AI on an empty diff" (Staging Model) and wastes an AI call on noise the user explicitly excluded. The two computations are logical aspects of one fact and should not be able to disagree; the safety check measures a different set than the set actually generated from.
  RECOMMENDATION: Make the emptiness decision derive from the SAME exclusion-filtered source the AI consumes, so the two cannot drift. Either apply excludePathspecs to the wouldStageNothing probes (so preflight measures the post-exclusion would-be-staged set), or move the emptiness verdict into the generator/L1 itself (compute the post-exclusion diff once, treat empty-after-exclusion as the same fail-loud "nothing to commit" case keyed on the actual tree state). The single source of truth is the post-diff_exclude would-be-staged content; both "fail loud on empty" and "generate from this" must read it.

- FINDING: The Committer seam is documented and named as the commit sink but is actually the general lock-resilient mutation sink (staging, commit, AND push)
  SEVERITY: medium
  FILES: internal/commit/run.go:142-156 (Committer interface doc), internal/commit/run.go:171-174 (Deps.Committer field doc), internal/commit/run.go:836-852 (stageForMode), internal/commit/run.go:907-924 (pushAfterCommit)
  DESCRIPTION: The interface doc says "the bare commit's `git commit -F -` flows through it" and the Deps.Committer field doc says "The commit — the bare path's ONLY mutation — flows through it." As shipped, three distinct git mutations route through Committer.Mutate: `git add -u`/`git add -A` (stageForMode), `git commit -F -` (createCommit), and `git push` (pushAfterCommit). The seam's contract is correct (every mutation needs the git_safe lock wrapper) but its name and doc under-describe it, framing it as commit-only when it is the lock-resilient MUTATION seam for the whole verb. This is the kind of narrow contract description that reads correctly in its original context (Phase 1 bare commit) but becomes misleading once staging/push were layered on, inviting a future reader to assume staging/push bypass the wrapper or to add a parallel push/stage path. The release engine's equivalent is named Mutator for exactly this reason.
  RECOMMENDATION: Rename Committer -> Mutator (mirroring engine.ReleaseDeps.Mutator, which it already says it mirrors) or at minimum rewrite the interface and field docs to state it is the lock-resilient sink for ALL of commit's git mutations (stage, commit, push), not solely the commit. The interface method is already the generic Mutate(ctx, stdin, name, args...), so only the type name and the two doc comments need to match the as-built scope.

- FINDING: The cmd-layer commit wiring (runCommit) has no direct test, leaving the production seam assembly uncovered
  SEVERITY: low
  FILES: cmd/mint/main.go:280-329 (runCommit), cmd/mint/dispatch_test.go:13 (only classifyCommand is tested)
  DESCRIPTION: internal/commit has strong end-to-end coverage of Run (real Generator + real git.Mutator over a single FakeRunner). The cmd-layer glue that constructs that thread — DetectStartupSignals -> StdinInteractive threading, NewForStartup, git.NewMutator(r) as the Committer, the flag->Deps mapping (Staging/NoAI/Yes/Push), and the exitCode mapping of commit's plain sentinels (errGateAborted/errPushFailed are NOT *engine.AbortError, so they fall through to generic exit 1) — is exercised only by classifyCommand routing tests. A regression in the StdinInteractive plumbing or the Push/NoAI field wiring would not be caught. NOTE: this matches the project-wide pattern (runRelease, runRegenerateSingle/All also wire NewExecRunner and are not directly tested at the cmd layer; only runVersion and the pure-ish resolveRegeneratePublisher are), so it is a consistent convention rather than a commit-specific regression — flagged low for completeness, not as a divergence.
  RECOMMENDATION: Optional and only if cheap: extract the flag-struct-to-Deps mapping (and the StdinInteractive/exitCode decisions) into a small pure helper that can be unit-tested without NewExecRunner, the way resolveRegeneratePublisher isolates the regenerate publisher decision. Otherwise accept it as consistent with the existing cmd-layer test boundary.

SUMMARY: The orchestrator's control flow (three fallback triggers, four gate choices, the routing sentinels) and the consumer-side seams (Committer/Transport/Deps, the shared OpenEditor roundtrip, the single pushAfterCommit reuse) compose cleanly and the L1/L2/L3 layering is sound. The one high-impact issue is a latent correctness gap: the empty-staging preflight and the AI's L1 source diff are parallel computations that apply different diff_exclude filtering, so an all-excluded staged set can reach the AI as an empty diff — they should derive from one exclusion-filtered source. A medium naming/doc drift on the Committer seam (commit-only doc vs. stage/commit/push usage) and a low-severity cmd-layer wiring test gap round it out.

- FINDING: Regenerate gh-auth gate keys off the resolved target, not the resolved publisher — diverges from the forward path's downgrade behaviour
  SEVERITY: medium
  FILES: internal/engine/regenerate.go:59-64, internal/engine/regenerate.go:99-103, cmd/mint/main.go:183-205, internal/engine/release.go:556-576
  DESCRIPTION: The forward spine resolves the publisher FIRST and runs CheckGhAuth only `if publisher != nil` (release.go:570) — a provider-unresolved downgrade (non-github / no-remote origin) warns and skips the gh-auth gate entirely, so the run is never stranded on a gh-auth failure for a release it was never going to publish. The regenerate path inverts this seam: the cmd layer resolves (and downgrades to a nil publisher with a warn) in resolveRegeneratePublisher BEFORE calling RegenerateRun, but RegeneratePreflight's gh-auth gate is then selected purely from target.writesProvider() (regenerate.go:59-64), with no knowledge of whether the publisher actually resolved. So a `regenerate --reuse`/`--target release` run on a non-github origin first warns "downgrade, provider skipped", then RegeneratePreflight still runs gh-auth and can abort the run — even though the provider write will be nil-guarded and skipped anyway. The two verbs handle an unresolvable provider with opposite outcomes (forward: silent skip of gh-auth; regenerate: gh-auth can still abort). The spec's "reuse must run gh-auth (a dead gh auth is the usual heal reason)" rationale assumes a resolvable provider; on a downgraded run there is no provider to authenticate against, so running gh-auth there is the same stranding the forward path's "gate a selected driver only" design exists to prevent. The seam is the gate-set selector consuming target where the forward path consumes publisher != nil.
  RECOMMENDATION: Thread the resolved-publisher fact (or its presence) into the regenerate gate-set selection so CallsProvider is false on a downgraded run — i.e. derive CallsProvider from `target.writesProvider() && publisher != nil`, mirroring the forward path's `if publisher != nil` guard. Since the cmd layer already resolves the publisher before dispatch, pass that result into the gate-set construction rather than re-deriving the gate set from the bare target inside RegenerateRun/RegenerateAllValidated. This makes the downgrade behaviour identical across both verbs and removes the chance of aborting a regenerate that has already downgraded.
- FINDING: Forward spine re-implements the warn-and-downgrade publisher branching that the new shared engine.ResolvePublisher already owns
  SEVERITY: low
  FILES: internal/engine/release.go:556-566, internal/engine/release.go:928-939
  DESCRIPTION: This phase introduced engine.ResolvePublisher (release.go:928) to consolidate the regenerate cmd sites' resolve-and-discard into one shared helper that applies the canonical "ErrProviderUnresolved → warn + nil-publisher; other error → surface; else driver" branching. But the forward engine.Release still hand-rolls the same three-way switch inline (release.go:559-566) over the raw resolvePublisher, because it needs the unresolved/error legs to route through surfaceAndUnwind (carrying made-state) rather than the helper's surface. The result is the warn-and-downgrade decision now living in two places that must be kept in lockstep — exactly the drift risk the helper was extracted to remove. The comment at 911-927 even asserts ResolvePublisher "mirrors the forward spine's choice precisely", which is true only by manual synchronisation, not by construction.
  RECOMMENDATION: Factor the shared decision (is-unresolved / is-error / driver) into one classifier the forward spine and ResolvePublisher both consult, letting each caller supply its own surfacing (surfaceAndUnwind vs surface) and downgrade-warn site. The branching predicate (errors.Is(err, ErrProviderUnresolved) vs other error) should be defined once so the two paths cannot diverge on which errors downgrade vs abort.
- FINDING: RunInteractive bypasses translateRun, so a context deadline/cancel during the editor hand-off is masked as a plain ExitError
  SEVERITY: low
  FILES: internal/runner/exec_runner.go:128-149, internal/runner/exec_runner.go:99-108
  DESCRIPTION: This phase hardened translateRun to inspect ctx.Err() before the ExitError branch (exec_runner.go:106) precisely so a SIGKILL'd child under a cancelled/expired context surfaces as a wrapped DeadlineExceeded/Cancelled rather than "signal: killed" — the property the AI transport's classifyFatal relies on. But RunInteractive (the editor seam) was left with its own hand-rolled translation (exec_runner.go:136-148) that does NOT consult ctx.Err(): a cancellation during the editor launch surfaces as a bare wrapped ExitError, so errors.Is(err, context.DeadlineExceeded) is false on that path. There is no timeout consumer of the editor launch today, so it is latent — but it is an inconsistent seam: two CommandRunner methods translate the identical exec outcome by different rules, and the editor launcher does flow under the same signal-cancellable ctx threaded from main. A future timeout or cancel handler around the editor hand-off would silently misclassify.
  RECOMMENDATION: Have RunInteractive route its non-nil run error through the same ctx-aware translation step (the ctx.Err()-first ordering), even though it captures no pipes — extract the not-found / ctx-cancel / wrap sequence so both pipe-capturing and interactive launches classify a cancelled child identically. This keeps the CommandRunner seam's error contract uniform across all three exec entry points.
SUMMARY: The Phase 10 seams largely compose well — the shared regenerate preflight, ResolvePublisher consolidation, []byte stdin, and context.WithoutCancel recovery are clean and well-tested. The one structural concern is that the regenerate gh-auth gate is selected from the bare target rather than the resolved publisher, giving the regenerate verb the opposite downgrade behaviour to the forward path; two lower-severity items (duplicated forward downgrade branching, RunInteractive's non-uniform error translation) are latent consistency gaps.

  SEVERITY: high
  FILES: internal/ai/transport_test.go:266-298 (TestTransport_Generate_RealDeadlineKillIsNonRetriedTimeout), specifically the 300ms deadline at :280 and the marker assertion at :291-297
  DESCRIPTION: The test forks a shell subprocess (`#!/bin/sh\nprintf x >> marker\nsleep 5`) under a fixed 300ms wall-clock Config.Timeout, then asserts the marker file contains exactly one byte to prove the subprocess ran exactly once (no retry). This makes the assertion depend on the forked process being able to fork + start sh + execute `printf` within 300ms. REPRODUCED: the test passes 5/5 in isolation but FAILS reliably under CPU contention — `go test -race ./internal/... ./cmd/...` and `go test -race ./internal/engine/... ./internal/git/... ./internal/runner/... ./internal/ai/... ...` both produce `reading invocation marker: ... no such file or directory` because the 300ms deadline SIGKILLs the subprocess before it writes its byte. The test's own comment claims 300ms is "generous enough... a tighter budget races the SIGKILL against process startup and flakes" — but under the race detector + parallel-package load that is exactly what happens. This violates golang-testing best-practice #3 ("Tests MUST NOT depend on execution order — each test MUST be independently runnable") and the code-quality determinism principle; it is the worst class of flake because it passes on a quiet dev machine and fails in busy CI. The timeout-not-retried behaviour is already proven deterministically by the sibling FakeRunner test (TestTransport_Generate_DoesNotRetryTimeout, :226), so the genuinely-new coverage here is only "a REAL exec.CommandContext deadline kill classifies as ErrTimeout" — which does not require asserting the marker side-effect at all.
  RECOMMENDATION: Decouple the assertion from the wall-clock startup race. The `errors.Is(err, ai.ErrTimeout)` check (:284) already proves the real-kill path classifies correctly and is timing-robust (the 5s sleep guarantees the 300ms deadline fires regardless of load). Drop the marker-file write/read side-effect (the script becomes just `sleep 5`, removing the marker setup and assertion), OR — if "exactly one invocation" must be asserted against a real subprocess — have the script append its byte BEFORE any sleep AND give the deadline enough headroom that startup cannot lose the race (e.g. a 2s deadline against a 30s sleep), accepting the added test latency. Either way the test must not gate a correctness assertion on a subprocess winning a fork-vs-SIGKILL race against a sub-second deadline.

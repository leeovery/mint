- FINDING: Regenerate FRESH path does not short-circuit a degenerate (empty/all-excluded) diff before the AI
  SEVERITY: medium
  FILES: internal/engine/regenerate_fresh.go:56-71, internal/notes/generate.go:102-108, cmd/mint/regenerate_run.go:64, cmd/mint/regenerate_all.go:89
  DESCRIPTION: The spec's "Degenerate release" guard (specification.md:258-262) states mint "does not call the AI" when the post-diff_exclude diff is empty/whitespace-only — "an empty diff is the one input it will reliably hallucinate on" — and writes the StubBody instead. On the FORWARD path this is enforced by notes.Selector.SelectBody, which runs IsDegenerate(diff) before invoking the AI generator (internal/notes/select.go:160-161). The regenerate FRESH path does NOT route through SelectBody: RegenerateFreshBody calls generator.GenerateFromRange directly, which assembles the vX-1..vX range diff and proceeds straight to CheckDiffSize then transport.Generate (internal/notes/generate.go:102-108, 136-157) with no IsDegenerate check anywhere in the chain. grep confirms IsDegenerate/StubBody are referenced only in the forward selector and tests — never in the regenerate engine or cmd code. This path is MORE exposed than the forward path: its range always contains mint's release-bookkeeping commit, so a version whose only non-excluded change was the bookkeeping (CHANGELOG.md + plain-mode version_file) yields an empty post-exclusion fresh diff that reaches the AI — exactly the "re-tag with no source change" / "every changed file fell under exclusion" cases the spec enumerates (specification.md:260). The --all skip-and-continue safety net does NOT cover this: a degenerate diff is not an AI *failure* (the AI returns a hallucinated body on success), so it is never intercepted as a per-version skip. The regenerate section (specification.md:504) says fresh re-diffs "with the same diff_exclude tiers + Change Map as the forward path" without explicitly re-listing the degenerate guard — so there is some spec ambiguity about whether fresh must inherit it. But the underlying intent ("never run the AI on an empty diff — it will hallucinate") is path-agnostic and the fresh path is precisely where a degenerate range is most likely, so this reads as a genuine under-implementation rather than a deliberate exclusion. No test exercises an empty/all-excluded fresh diff.
  RECOMMENDATION: Apply the same notes.IsDegenerate short-circuit the forward path uses: after AssembleRange produces the post-exclusion range diff and before the AI call, if IsDegenerate(diff) return notes.StubBody() with no AI invocation — mirroring SelectBody's degenerate branch. Cleanest reuse is to add the check inside GenerateFromRange (or a thin wrapper RegenerateFreshBody calls) so both single (regenerate_run.go:64) and batch (regenerate_all.go:89) fresh producers inherit it, keeping "degenerate -> stub, never the AI" a single shared rule rather than forward-only. Add a fresh-path test asserting an empty/all-excluded vX-1..vX diff yields StubBody with no transport call. If the team instead concludes the spec intends fresh-regenerate to be exempt, record that decision explicitly — but the spec's stated rationale argues against exemption.
SUMMARY: One genuine spec-conformance gap: the regenerate fresh path bypasses the degenerate-diff guard that the forward path enforces via SelectBody, so an empty/all-excluded vX-1..vX range diff can reach the AI — the exact hallucination case the spec's degenerate-release section forbids. All other checked decision points (tag grammar, --set-version strictly-greater gate and exact messages, version-file pattern-mismatch abort, provider downgrade-not-fail, regenerate axis-contract and reuse/changelog/target error messages, commit subjects, AI retry/timeout semantics, init shim, version surface) conform to the specification and Go conventions.

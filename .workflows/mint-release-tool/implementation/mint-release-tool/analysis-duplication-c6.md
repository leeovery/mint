- FINDING: Single-version and batch cmd-layer body/regenerator producers are near-identical pairs
  SEVERITY: medium
  FILES: cmd/mint/regenerate_run.go:59-66, cmd/mint/regenerate_run.go:75-82, cmd/mint/regenerate_all.go:95-102, cmd/mint/regenerate_all.go:110-117
  DESCRIPTION: Four closure-builders implement two pieces of logic twice. The body producers (newRegenerateBodyProducer vs newBatchBodyProducer) both branch `if source == RegenerateSourceReuse { return engine.ReadReuseBody(...) } return engine.RegenerateFreshBody(...)`. The regenerator producers (newRegenerateRegeneratorProducer vs newBatchRegeneratorProducer) both branch `if reuse { return nil } return engine.RegenerateFreshRegenerator(...)`. The only difference is the single-version variants close over one fixed version.Resolution while the batch variants take the Resolution as a parameter. The dispatch rule (reuse→read, fresh→generate; reuse→no regenerator, fresh→regenerator) is the same decision authored twice in each pair. If the source→producer mapping ever changes (e.g. a third source, or a different fresh transport wiring), all four sites must be kept in sync independently.
  RECOMMENDATION: Collapse each pair to one per-Resolution helper and have the single-version site bind its fixed Resolution. e.g. a single `regenerateBodyFor(ctx, r, cfg, root, source, res)` and `regenerateRegeneratorFor(r, cfg, root, source, res)` keyed on the Resolution; newRegenerateBodyProducer/newBatchBodyProducer then become trivial closures over the shared function (the single-version one capturing `res`, the batch one threading it through). This removes the duplicated source-branch from two sites to one each.
- FINDING: AI-transport resolution duplicated between forward and fresh-regenerate paths
  SEVERITY: low
  FILES: internal/engine/release.go:849-854, internal/engine/regenerate_fresh.go:126-131
  DESCRIPTION: aiTransport (forward) and resolveFreshTransport (regenerate) are the same logic: "return the injected transport when non-nil, else build the production ai.Transport via `ai.NewTransport(r, ai.Config{AICommand: cfg.AICommand})`". Both re-default the empty ai_command identically through NewTransport. The bodies differ only in how the runner is reached (deps.Runner vs a passed `r`). The `ai.Config{AICommand: cfg.AICommand}` literal also appears verbatim at both sites. The comments at both helpers explicitly note they "mirror" each other — a copy-paste-drift smell across the task boundary.
  RECOMMENDATION: Extract one `resolveTransport(r runner.CommandRunner, injected notes.Transport, cfg config.Config) notes.Transport` in the engine package and have both aiTransport and resolveFreshTransport delegate (the forward one passing deps.Runner). This also centralises the single `ai.Config{AICommand: cfg.AICommand}` construction.
- FINDING: Forward notes ExcludeConfig literal duplicates freshExcludeConfig
  SEVERITY: low
  FILES: internal/engine/release.go:741-745, internal/engine/regenerate_fresh.go:112-118
  DESCRIPTION: The forward path (resolveBody) inline-constructs `notes.ExcludeConfig{Globs: cfg.DiffExclude, VersionFile: cfg.Release.VersionFile, VersionPattern: cfg.Release.VersionPattern}` while the regenerate path has already factored the identical construction into freshExcludeConfig(cfg). Because the forward site is an inline literal rather than a call to the shared helper, the two can drift if a new exclude tier is added (a fourth field would be silently omitted on whichever site is missed).
  RECOMMENDATION: Have resolveBody call the existing freshExcludeConfig(cfg) instead of inlining the literal, so the config→ExcludeConfig projection lives in exactly one place. No new code needed, only redirecting the forward site to the existing helper.

The "mint: " prefix and the usage exit code are a single convention reproduced by hand at each call site.

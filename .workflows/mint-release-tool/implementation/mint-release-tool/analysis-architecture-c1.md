- FINDING: `r` (regenerate-with-context) review choice is rendered on the regenerate fresh gate but has no production backing — guaranteed abort
  SEVERITY: high
  FILES: cmd/mint/main.go:122-129, internal/engine/regenerate_write.go:186, internal/engine/regenerate_batch.go:310, internal/engine/release.go:1337-1350, internal/presenter/gate.go:143-159
  DESCRIPTION: The fresh regenerate path runs the four-choice notes-review gate (`presenter.NotesReviewGate()`, which renders `[y] [n] [e] [r]`) via `gateRegenerate -> reviewGate(ctx, p, deps.Editor, deps.Regenerator, notes.KindNormalAI, ...)`. That gate offers `r` (regenerate with context) for KindNormalAI. But the regenerate cmd entry point (`runRegenerate`, main.go:122) builds `engine.ReleaseDeps` WITHOUT a `Regenerator`, and — unlike the forward `Release` orchestrator, which constructs a per-run regenerator via `perRunRegenerator(...)` — the regenerate engine path (`RegenerateRun`/`RegenerateWrite`/`RegenerateAll`) never builds one. So `deps.Regenerator` is nil on every interactive `mint release regenerate <ver> --fresh` (and `--all --fresh`). When the user picks `r`, `regenerateBody` hits `regen == nil` and surfaces `errRegeneratorUnavailable` ("regenerate chosen but no regenerator is configured") and aborts the run. The engine comment calls this "a misconfiguration that should never reach production" — but for the regenerate verb it reaches production by construction. This is a seam defect: a declared, rendered, user-reachable presenter choice is wired to a stub on this path. The asymmetry is sharp — the `e` (edit) choice on the SAME gate IS wired for regenerate (main.go:128 sets `Editor`), so `e` works and `r` does not, even though the spec (lines 448/454/533: "fresh regeneration runs the same notes-review gate", "[r] regenerate with context") presents them as peers.
  RECOMMENDATION: Wire a per-run Regenerator on the regenerate fresh path the same way the forward path does. The fresh body producer (`newRegenerateBodyProducer` / `newBatchBodyProducer`) already builds the notes Generator from `(r, cfg, root, res)`; bind its `GenerateWithContext` over the resolved diff range into `deps.Regenerator` (or pass a per-run regenerator into `RegenerateWrite`/`gatePerVersion`) so `r` re-runs the same fresh AI path with the one-time context appended. Alternatively, if `r` is genuinely out of scope for regenerate, build the fresh regenerate gate from a y/n/e variant (not `NotesReviewGate()`) so the choice is never offered — but the spec lists `r` for fresh regenerate, so wiring it is the spec-faithful fix.

- FINDING: Release orchestrator never emits per-stage progress events (StageStarted / StageSucceeded) — the spinner is dead and stage narration is absent
  SEVERITY: medium
  FILES: internal/engine/release.go:279-617, internal/engine/regenerate_interactive.go:134-170, internal/engine/editor.go:103-104, internal/presenter/presenter.go:64-79, internal/presenter/spinner.go:29-30
  DESCRIPTION: The Presenter contract (owned by the CLI Presentation spec, consumed by this spec per Dependencies line 681: "The release run emits semantic events to a `Presenter`") declares `StageStarted` and `StageSucceeded`, and the presentation side built+tested a blocking-stage spinner that animates between a blocking `StageStarted` and its completion. The engine orchestrator emits `RunStarted`, `ShowPlan`, `ShowNotes`, `Prompt`, `Warn`, `StageFailed`, `Unwound`, and `RunFinished` — but `StageStarted` and `StageSucceeded` have ZERO production call sites anywhere in `internal/engine` or `cmd/` (they are not even referenced in engine tests; only the presenter's own golden-transcript test drives them to exercise rendering). Consequences that compose poorly across the seam: (1) a real `mint release` run shows no per-stage narration — no "version ok", "preflight ok", "generating notes...", "push ok" — only the plan, the notes block, the gate, and the final line; (2) the long stages this spec defines (AI notes ~60s, `pre_tag` build hooks, the atomic push) run with no progress indicator, which is the exact use case `StageStart.Blocking` and the spinner were built for; (3) the editor's `SuspendSpinner`/`ResumeSpinner` bracket (editor.go:103-104) presumes a live spinner to suspend, but since no `StageStarted` ever fires there is never an active spinner — the bracket is permanently a no-op. The presenter half of the seam was built to a contract the engine half never drives.
  RECOMMENDATION: Have the release (and regenerate) orchestrators emit `StageStarted`/`StageSucceeded` around the stages they already sequence — at minimum the blocking ones (notes generation, `pre_tag` hook, push), with `Blocking:true` so the spinner activates, and ideally the cheap read-only gates (version, preflight) for completion narration. The payload structs (`StageStart`, `StageSuccess` with engine-measured `Elapsed`) already exist; this is wiring emit calls into the existing stage sequence, not new structure. Once driven, the editor's suspend/resume bracket becomes live as intended.

- FINDING: `--reuse --all` reads each tag's annotation body twice (redundant `git for-each-ref` per version)
  SEVERITY: low
  FILES: internal/engine/regenerate_batch.go:213-225, internal/engine/regenerate_reuse.go:41-67, cmd/mint/regenerate_all.go:84-91
  DESCRIPTION: On the batch reuse path, `processOneVersion` calls `ReadTagBody(ctx, deps.Runner, res.Tag)` (batch_213) purely for the `hasBody` skip-or-continue decision, discarding the body it returns. It then calls `req.ProduceBody(...)` (batch_225) which, for reuse, routes to `newBatchBodyProducer` -> `engine.ReadReuseBody` -> `ReadTagBody` AGAIN — so `git for-each-ref --format=%(contents:body) refs/tags/<tag>` runs twice per version for the entire backfill. `ReadTagBody` already returns `(body, hasBody, err)` in one call, so the first read carries everything the second produces; the second read recomputes identical bytes. It is not a correctness bug (the tag is immutable between the two reads), only a redundant git invocation per version that grows linearly with the backfill. The single-version reuse path does NOT have this (it calls `ReadReuseBody` once and never separately checks has-body), so the duplication is batch-only. This is a missed-composition seam: the batch's has-body gate and its body production are logically the same read split into two.
  RECOMMENDATION: Collapse to one read on the batch reuse path — keep the body from the `ReadTagBody` call at batch_213 and pass it through for reuse (branch reuse vs fresh in the loop and reuse the already-read body), or have the reuse `ProduceBody` accept the pre-read body. Minor; worth doing only when the file is next touched.

# Research: Release-Notes Quality

Can AI release-note quality be lifted beyond what a raw textual diff allows ‚Äî especially for large releases where big diffs summarise to mush? This thread explores enriching the *input* to the AI (AST / semantic structure / other signals) rather than further tuning the prompt and output format.

## Starting Point

What we know so far:
- **Prompted by**: Suspicion that AI release-note quality can be lifted beyond a raw textual diff, especially for large releases where big diffs summarise to mush (echoes the first discussion's `max_diff_lines` cost/quality note).
- **Already knows**: The first discussion tuned the AI prompt and output format but always fed the AI a raw textual diff. This thread explores enriching the *input* instead. Framed explicitly as speculative ‚Äî an open "what is possible" question.
- **Starting point**: Technical feasibility ‚Äî feeding AST or richer semantic structure of the change (or some other signal) to produce more accurate, meaningful notes.
- **Constraints**: none

## Prior Context (from knowledge base)

What's already settled that this thread builds on:

- **First discussion ‚Äî AI release-notes quality (parked child).** Settled the *prompt/output* side: diff-exclusion tiers (`CHANGELOG.md` always; `version_file` strategy-aware; `diff_exclude` globs), `max_diff_lines` (default 50000) reframed as a **cost + quality** guard ("a huge diff is slow, costly, and summarises to mush"), and a default format (TL;DR one-liner + emoji-headed sections, bolded notable features, no-preamble rule) plus two-knob prompt config (`notes_context` inject / `notes_prompt` override). **All of this always feeds the AI a raw textual diff.** This research thread is the parked child that asks whether enriching the *input* does better.
- **Commit-command engine boundary ‚Äî names this thread directly.** The AI message engine is a **three-layer split**: L1 = context builder (git-aware, produces "the content to describe"), L2 = AI message engine (**git-unaware, content-agnostic** ‚Äî "context in, message out"), L3 = per-verb glue. The load-bearing property: *"The input being a diff is incidental; L2 just sees 'content.'"* Explicitly: *"this is what makes the separate **release-notes-quality** research thread cheap: enriching the input (AST/semantic signal instead of a raw diff) swaps L1's output with **zero change to L2**. The boundary was chosen partly because it absorbs that future work."*

**Implication for this research:** the architectural home is already carved out. Anything this thread proposes lives inside **L1 (the context builder)** ‚Äî it changes *what content is assembled*, not the engine, prompt-ownership model, or sinks. That's the lens to keep.

## Framing: the failure is salience, not absence

The user's prompting instinct: on big releases the AI sometimes "glosses over" a large feature and over-focuses on smaller items. Probing this, the failure shape is **misallocated attention**, not **missing data** ‚Äî the big change is *in* the diff; the summary budget gets spent elsewhere. This splits the solution space:

- **Missing data** ‚Üí fix is *feed more / feed differently* (AST/semantic extraction lives here).
- **Salience** ‚Üí fix is *signal what matters*; feeding more raw diff can make it **worse** (more noise, closer to the "mush" ceiling).

A raw unified diff is **anti-salient**: a 3-line config tweak and a 400-line new subsystem both render as "a hunk with `+` lines." The lone magnitude signal (`git diff --stat`) was present in the old script but is a footnote the model can ignore. Nothing in the input says "*this* is the headline."

## Empirical pass ‚Äî agentic-workflows real notes vs real diffs

Examined CHANGELOG + tag annotations (generated by the old `release` script) against the actual diffs. Findings:

1. **The notes are already fairly good.** v0.5.0 (progress-decay), v0.4.13 (universal discovery), v0.3.0 (knowledge base) ‚Äî all big releases, all got accurate notes with the real feature as the lead. Much of the original "mush" feeling is attributable to the **old flat format** (no TL;DR/sections, equal-weight list) ‚Äî which *this epic already fixed*. Honest read: part of the motivating problem is solved by format work, not input enrichment.

2. **The real input problem is artifact noise, not source weighting.** A large share of diff *line-count* in this repo is high-volume, low-importance non-source:
   - v0.4.13: the single biggest file in the diff was `discovery-map/phase-17-entry-ux-redesign.md` ‚Äî **1157 lines, ~20% of total churn** ‚Äî a *planning doc*, not source.
   - v0.3.0: ~‚Öì of churn was test files; more was `knowledge-base/*.md` docs.
   - v0.5.0: `ideas/*.md` files sat in the diff alongside the code.
   This noise both competes for AI attention *and* inflates toward `max_diff_lines` (the cost/mush guard). The decided `diff_exclude` handles *known* build artifacts (bundle, lockfiles) but not planning docs / ideas / tests.

3. **AST/semantic is a poor fit for *this* repo.** agentic-workflows is markdown skills ‚Äî there is almost no code AST to extract. AST enrichment would help a code-heavy repo (Go/TS) but does little for the user's primary use case. This is a **generality question for mint**: who is the enrichment for?

### Threads opened (not yet pulled)

- **A. Artifact-aware noise reduction** ‚Äî cheapest, highest-leverage *here*. Overlaps already-decided machinery (`diff_exclude` + `max_diff_lines`). Question: convention/heuristic vs config ‚Äî should mint treat `tests/`, planning/idea docs as lower-priority *signal* rather than all-or-nothing exclusion?
- **B. Magnitude/salience hint** ‚Äî prepend a ranked "largest changes by area/file" summary so the model has an explicit headline signal, cheaply (no AST). Directly targets the salience failure.
- **C. Commit-message signal** ‚Äî the `vX-1..vX` range's commit subjects carry human intent. Old script *ignored* them (`ignore 'wip' commits`) ‚Äî but as the repo adopts `mint commit` ‚Üí Conventional Commits, subjects become high-signal structure. Tension: wip noise vs structured intent.
- **D. AST/semantic structure** ‚Äî the original hypothesis. Real for code-heavy repos (new exported symbols, new files, public API surface = headline). Weak for markdown. Most expensive.

## Pivotal reframe: generality inverts the hypothesis

**User intent (decisive):** mint is for *every* repo they manage now or in future ‚Äî markdown (agentic-workflows), Go (Portal, Tick, **mint itself**, which dogfoods/releases itself), Bash (Stitch), SDKs, anything built later. Generality is not a nice-to-have; it's the tool's reason to exist.

This **inverts the starting (AST) hypothesis.** AST is the *least* language-agnostic enrichment possible: the "what's the headline" extraction query is per-language (Go's exported-identifier ‚âÝ Bash, ‚âÝ markdown). Tree-sitter unifies *parsing* (one lib, many grammars) but not *extraction*. So D is in active tension with mint's defining requirement ‚Äî it ranks low not just on cost but on fit.

**Generality promotes the language-blind signals ‚Äî all free from git:**
- **C. Commit intent** ‚Äî every repo has commit messages. Language-agnostic. Conditional on commit hygiene (wip noise vs structured intent ‚Äî the old script ignored commits *for this reason*).
- **Structural signal (new sub-thread, call it E)** ‚Äî `git diff --name-status`: new files, new directories, a whole new package/skill folder appearing = a "headline" signal in *any* language, for free.
- **B. Magnitude** ‚Äî line/file counts; language-blind but distorted by artifact noise unless A runs first (review F5/F10).

**Unique-to-mint kicker ‚Äî mint owns both ends of the pipe.** mint is multi-verb: `mint commit` (just decided ‚Üí Conventional Commits) authors the very commit history `mint release` later consumes. So mint can *improve the signal it ingests* ‚Äî better commits in ‚Üí better notes out. No single-purpose release tool can do this because it doesn't author your commits. This makes **thread C compounding, not just additive**, for repos that adopt `mint commit`. (Graceful degradation still required for repos that don't ‚Äî Stitch, legacy history.)

**Artifact-noise (thread A) ‚Äî largely closed by already-decided config.** User confirms: `.gitignore` is inherently respected (tag-to-tag diff can only contain tracked files ‚Äî matches prior `diff_exclude` decision), and `diff_exclude` lets each repo mark checked-in-but-noise dirs/files (planning docs, ideas) as excluded from the diff. So A is *use existing config*, not new research. **Residual nuance:** exclusion is all-or-nothing, but "added test coverage for X" is sometimes a legit note ‚Äî a gap between *exclude entirely* and *present-but-deprioritised*. Flagged, not urgent.

### Cost lens (review F3) applied to the reframe
The language-blind signals (C, E, B) are **near-zero marginal cost** ‚Äî git already computes them (`git log`, `git diff --name-status`, `--stat`). AST (D) needs parsing infra + per-language extraction. For a *general* tool, cheap-and-general dominates expensive-and-per-language on every axis the parent discussion cares about (cost is explicitly first-class via `max_diff_lines`).

## Prior art (deep dive ‚Äî commit-driven tooling landscape)

Investigated `release-please`, `semantic-release`, `git-cliff`, `conventional-changelog`, GitHub auto-notes / `gh release`. Strategic findings:

- **The entire established ecosystem is commit-message-only regex ‚Äî no diff, no AI.** Parse commit text ‚Üí group by Conventional-Commit `type` ‚Üí render a template. mint's current design (diff is truth, commits ignored) is the **exact inverse** of every shipped tool. So the diff+commit blend is genuinely novel territory, not a solved pattern.
- **Conventional Commits is the universal backbone.** One `type` token drives *two* outputs: changelog section (`feat`‚ÜíFeatures, `fix`‚ÜíBug Fixes) **and** semver bump (`fix`‚Üípatch, `feat`‚Üíminor, `!`/`BREAKING CHANGE:`‚Üímajor). Identical across all tools. Proven, language-agnostic. `scope` is the secondary axis (monorepo routing).
- **GitHub auto-notes / `gh --generate-notes` are PR-driven, not commit-driven** ‚Äî unit is the merged PR (title + labels). **Degrades to near-empty without PRs** (cli/cli#6159). The user's solo, direct-to-default-branch repos (agentic-workflows, Stitch) have few/no PRs ‚Üí GitHub's native generator is structurally useless for them. *This is a concrete reason mint exists.*
- **"A changelog is not a commit log" (the central quality complaint).** Regex tools render one-commit-one-line and **structurally cannot synthesise across commits** ‚Äî that's the architectural ceiling on narrative quality. A real changelog entry is "a noteworthy difference, often across multiple commits," written for *users* not developers. **mint's AI narrative layer is precisely what escapes this ceiling** ‚Äî a key differentiator.
- **The hybrid (diff + commit-intent + AI) exists only as fragmented frontier pieces ‚Äî nobody ships it as one general tool.** Classic tools = commits-only-no-AI; young AI tools pick *one* of diff-or-commits (e.g. AI-Changelog-Generator = diff-input; Changeish/deployHQ = commit-input). The synthesis argument ("diff says *what*, commit/PR says *why*") is stated in the literature but unpackaged. **git-cliff's `git cliff --context` ‚Üí JSON ‚Üí LLM seam** is the closest existing pattern and a ready-made reference schema (per-commit id/group/scope/body/footers/breaking/raw_message).
- **Industry has converged on language-blind signals; AST is absent from a decade of changelog tooling** ‚Äî for exactly mint's generality reason. Strong external corroboration that thread D ranks low on *fit*, not just cost.

### Strategic synthesis ‚Äî mint's unique position (deep-dive F7)
**Every surveyed tool consumes a commit history it did not author** ‚Üí their universal, unfixable weakness is GIGO (garbage commits ‚Üí garbage notes); they can only defensively filter/drop non-conforming commits. **mint is the only tool that sits on both ends of the pipe** (`mint commit` authors Conventional Commits; `mint release` consumes them). So mint can treat the commit history as a *designed* signal it produced, not an unreliable one to filter. This inverts the GIGO assumption that forces every competitor to either drop commits or fall back to the diff. **No single-purpose release tool can replicate this ‚Äî none of them write your commits.**

- **Borrow:** Conventional-Commits type‚Üísection/semver mapping; `scope`/path-filter attribution; git-cliff's structured-JSON-to-AI shape.
- **Don't borrow:** one-commit-one-line regex rendering (mint's AI layer already beats it); PR/label dependence (no PRs in solo repos); AST (industry avoids it for generality).
- **Graceful degradation required** for repos mint *doesn't* author commits for (Stitch, legacy history) ‚Äî fall back toward diff-truth; wip commits there are noise, not signal.

## Where this lands ‚Äî the converged picture

The thread converged. The enrichment shape is: **diff (ground-truth *what*) + structured commit-intent (the *why*/headline) + AI (cross-commit narrative synthesis)** ‚Äî all language-agnostic. Supporting cheap signals: **structural** (new files/dirs/packages via `--name-status`) and **magnitude** (`--stat`). De-prioritised: **AST/semantic** (industry avoided it for a decade for mint's exact generality reason; near-useless for the markdown dogfood repo). Already-handled: **artifact noise** (existing `diff_exclude` config + inherent `.gitignore` exclusion).

The motivating failure ("glosses over the big feature") is a **narrative/story failure**, not a *what-changed* failure ‚Äî and mint's AI layer is uniquely the mechanism that escapes the regex tools' "one-commit-one-line, can't synthesise across commits" ceiling.

**mint's structural advantage:** it owns *both ends of the pipe* (`mint commit` authors Conventional Commits; `mint release` consumes them) ‚Äî no competitor can, because none author your commits.

## Open Questions (for the discussion phase to decide)

These are **decisions**, deliberately not made here:

1. **Should `mint release` notes ingest commit data at all** ‚Äî reversing the old script's explicit "ignore commit messages"? If yes, **how much weight** does commit-intent carry vs the diff (diff as primary + commits as headline hint? commits as spine + diff as detail/fallback?).
2. **Which commit signal is highest-value** ‚Äî the `type`/`scope` *structure*, the *subject* as a human headline, or the *body* (where the "why" lives, which the classic tools mostly ignore)? Possibly unmined signal in commit bodies.
3. **Graceful degradation** for repos mint doesn't author commits for (Stitch, legacy history, wip-heavy histories) ‚Äî when does commit data help vs become noise? Default posture when commits are unconventional.
4. **Structural headline hint (thread E)** ‚Äî is "a whole new directory/package appeared" worth surfacing explicitly to the AI as a magnitude/salience hint? (Note: any line-count-ranked magnitude hint must run *after* `diff_exclude`, or the 1157-line planning doc ranks as the "headline" ‚Äî B depends on A.)
5. **Token-budget interaction** ‚Äî enrichment competes with the diff for the `max_diff_lines` budget on exactly the large releases this targets. A structured commit/structural summary is *cheap* (tens of lines) and could let mint *shrink* the diff (summary carries headline, diff carries detail). Design question for spec.
6. **`diff_exclude` granularity (minor)** ‚Äî exclusion is all-or-nothing, but "added test coverage for X" is sometimes a legit note. Gap between *exclude entirely* and *present-but-deprioritised*. Low priority.

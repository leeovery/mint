- FINDING: Three near-identical transport-wiring sites carry the same construction logic with no shared seam
  SEVERITY: low
  FILES: internal/engine/release.go:936-944 (aiTransport), internal/commit/run.go:780-788 (commitTransport), internal/engine/regenerate_fresh.go:135-143 (resolveFreshTransport)
  DESCRIPTION: All three production wiring sites are the same shape — `if injected != nil { return injected }; return ai.NewTransport(r, ai.Config{AICommand: cfg.AICommandFor(verb), Timeout: cfg.TimeoutFor(verb)})` — differing only in the verb constant (VerbRelease / VerbCommit / VerbRelease) and the test-seam field. The spec explicitly flags regenerate_fresh.go as the "easy miss" that must resolve through [release], and the correctness of that routing currently rests entirely on each site independently passing the right verb constant. There is no single helper that maps a `(runner, config, verb)` triple to an `ai.Config`, so the "thread both keys from the verb accessor, never zero-by-omission" contract is re-expressed three times rather than owned in one place. This is a seam-quality observation, not a DRY nit: the invariant the spec cares most about (regenerate rides [release]) is enforced only by convention + per-site tests, not by construction. NOTE: the NewTransport nil-panic guard already makes zero-by-omission unreachable, and each site has a dedicated white-box test pinning its verb, so the latent risk is contained — hence low severity.
  RECOMMENDATION: Consider a small shared constructor, e.g. `ai.NewTransportFor(r, cfg config.Config, verb config.Verb)` or a `config`-side helper returning a populated `ai.Config`, so the accessor-sourcing + verb selection lives once. The three sites would then differ only in the verb constant passed, making the regenerate-rides-[release] decision a one-token choice at a single, obvious call rather than a repeated pattern. Only worth doing if it does not force `config` to import `ai` (the spec mandates that decoupling survives) — a thin engine/commit-side helper, or `ai` accepting a `config.Config`, both preserve it; pick whichever keeps the import direction clean.

- FINDING: AICommandFor and TimeoutFor build their candidate lists with divergent shapes despite being sibling accessors
  SEVERITY: low
  FILES: internal/config/config.go:539-562 (AICommandFor), internal/config/config.go:595-623 (TimeoutFor)
  DESCRIPTION: The two layered accessors are presented as mirror-image siblings ("mirroring AICommandFor's structure") but assemble their candidate chains differently. AICommandFor selects the override into a local, then conditionally prepends it (`candidates := []string{c.AICommand, DefaultAICommand}; if override != nil { candidates = append([]string{*override}, candidates...) }`) — a slice rebuild that places a non-nil override first. TimeoutFor uses a fixed three-slot list with the raw (possibly nil) override always in slot 0 (`candidates := []*time.Duration{override, c.Timeout, &floor}`) and relies on the loop's nil-skip. Both are correct, but the asymmetry means a reader cannot transfer intuition from one to the other, and the AICommandFor prepend allocates a second slice purely to avoid carrying a nil/blank string in slot 0 — which the trim-and-skip loop would already discard. The structural difference is incidental, not motivated by the keys' differing value semantics (which justifiably differ in the loop body, not the list construction).
  RECOMMENDATION: Unify the candidate-list construction so the two accessors differ ONLY in their per-candidate rule (the switch body), not in how the chain is assembled. AICommandFor can use the same always-three-slot shape — `[]string{override-or-empty, c.AICommand, DefaultAICommand}` (an absent override contributes "" which the existing trim-skip already drops) — removing the conditional prepend and the extra allocation, and making the sibling relationship literal. This is a readability/consistency improvement on already-correct code; flag only because the two are explicitly documented as mirrors.

The only observations are a low-severity seam-quality point (three structurally-identical transport-wiring sites with no shared constructor, so the spec's "easy miss" regenerate routing is enforced by convention rather than construction) and a low-severity consistency nit in how the two sibling accessors assemble their candidate lists.
